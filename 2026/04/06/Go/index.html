Golang 中 make 和 new 的区别？

make 和 new 都是用于内存分配的内建函数，但它们的使用场景和功能有所不同：

make：

  • 用于初始化并分配内存，只能用于创建 slice、map、channel 三种类型。

  • 返回的是初始化后的数据结构，而不是指针。

new：

  • 用于分配内存，但不初始化，返回的是指向该内存的指针。
  • 可以用于任何类型的内存分配。

讲一下进程、线程、协程

  • 进程

进程是操作系统进行资源分配的基本单位，每个进程有独立的虚拟地址空间和资源边界。

它最大的价值是隔离性和稳定性：一个进程崩溃通常不会直接污染另一个进程，因此在安全性、故障隔离、独立部署方面优势明显。
代价是进程创建、销毁、切换都更重，进程间通信也不能像线程那样直接共享内存，通常要通过 IPC（管道、消息队列、共享内存、Socket 等）完成。

  • 线程

线程是操作系统调度的基本单位，属于内核态可见的执行流。

线程存在于进程内部，同一进程内的多个线程共享地址空间和大多数进程资源（如堆内存、文件描述符），但每个线程有自己的栈和寄存器上下文。

相比协程，线程切换需要内核参与，开销更高；相比进程，线程创建与切换又更轻量。

线程的优势是能够直接利用多核并行，尤其适合 CPU 密集型计算任务，但因为线程共享内存，开发中必须认真处理竞态条件、锁竞争和死锁问题。也就是说，线程在性能与复杂度之间提供了一个中间层：并行能力强于单线程模型，管理复杂度又明显高于协程的高层并发写法。

  • 协程

协程是用户态的轻量执行单元，核心特点是“由运行时调度而不是由操作系统内核直接调度”。

这意味着协程切换通常发生在用户态，切换成本低、内存占用小，适合承载大量高并发 I/O 任务，比如网络连接、RPC、消息处理、爬虫等。

协程本身更偏“并发抽象”，它可以让我们写出看起来顺序、实际高并发的代码，但协程是否真正并行执行，取决于底层有没有多个内核线程参与调度。协程间通信通常使用更高层的机制，比如 channel、await/async 这一类模型，以降低直接共享内存带来的锁复杂度。

Go 语言中，goroutine 就是协程，运行时用 GMP 模型把大量 G 映射到较少的 OS 线程上，所以能在较低资源成本下支持非常高的并发规模。

defer 的执行顺序是怎样的？defer 的作用或者使用场景是什么?

defer执行顺序和调用顺序相反，类似于栈后进先出(LIFO)

defer 的作用是：当 defer 语句被执行时，跟在 defer 后面的函数会被延迟执行。直到 包含该 defer 语句的函数执行完毕时，defer 后的函数才会被执行，不论包含 defer 语句的函数是通过 return 正常结束，还是由于 panic 导致的异常结束。 你可以在一个函数中执行多条 defer 语句，它们的执行顺序与声明顺序相反。

defer 的常用场景:

  • defer语句经常被用于处理成对的操作，如打开、关闭、连接、断开连接、 加锁、释放锁。

  • 通过defer机制，不论函数逻辑多复杂，都能保证在任何执行路径下，资源被释放。

  • 释放资源的defer应该直接跟在请求资源的语句后。

slice扩容机制

面试里可按三点展开：扩容会分配新底层数组并 copy 原数据；小容量常见近似 2 倍增长，大容量增长倍率下降（避免内存浪费）；扩容后旧切片和新切片可能不再共享底层数组。

关键点：

  • 扩容会分配新底层数组并 copy 原数据。
  • 小容量常见近似 2 倍增长，大容量增长倍率下降（避免内存浪费）。
  • 扩容后旧切片和新切片可能不再共享底层数组。

Go1.18及以后，引入了新的扩容规则：

当原slice容量(oldcap)小于256的时候，新slice(neWcap)容量为原来的2倍；原slice容量超过256，新slice容量newcap = oldcap+(oldcap+3*256)/4

map底层设计（map是不是并发安全的）

Go Map的底层实现是一个哈希表。它在运行时表现为一个指向 hmap 结构体的指针。

  • hmap 中记录了桶数组指针 buckets 、溢出桶指针以及元素个数等字段。
  • 每个桶是一个 bmap 结构体，能存储8个键值对和8个 tophash ，并有指向下一个溢出桶的指针 overflow 。
  • 为了内存紧凑，bmap 中采用的是先存8个键再存8个值的存储方式。

并发安全结论非常关键：map 不是并发安全的。

  • 多个 goroutine 并发读通常没问题（前提是没有写），但只要出现并发写，或者一边读一边写，就是未定义行为，常见会直接报 fatal error: concurrent map read and map write 或 concurrent map writes。
  • 并发场景用 sync.RWMutex 或 sync.Map。

sync.Map与普通map的区别

sync.Map有以下几个关键的特点：

  1. 并发安全：sync.Map内部使用了锁和其他同步原语来保证并发访问的安全性。sync.Map的设计目标是为了提供一个高效的并发安全的map，特别是在读多写少的场景下。
  2. 无需初始化：sync.Map不需要像内置map那样使用make函数初始化，可以直接声明后使用。
  3. 特殊的API：sync.Map提供了特定的方法如Load, Store, LoadOrStore, Delete, 和Range，而不是使用内置map的语法。

CSP是什么

CSP（Communicating Sequential Processes）强调“通过通信共享内存”，用消息传递降低共享状态竞争。

  • goroutine 负责执行
  • channel 负责通信
  • 目标是减少锁竞争和共享可变状态

在 Go 中是并发设计思想，不是“完全不用锁”。

讲一下 Go 中的 Channel

可直接说： channel 是 goroutine 间的类型安全通信管道，支持同步与缓冲两种模式。面试里可按三点展开：无缓冲：发送和接收必须同步配对；有缓冲：缓冲未满可直接发送，未空可直接接收；常与 select、context 组合做并发控制与退出。

关键点：

  • 无缓冲：发送和接收必须同步配对。
  • 有缓冲：缓冲未满可直接发送，未空可直接接收。
  • 常与 select、context 组合做并发控制与退出。

select 和 switch 的区别

可直接说： switch 做表达式分支，select 做 channel I/O 多路选择。面试里可按三点展开：select 只用于 channel 的发送/接收分支；switch 是普通控制流，与并发通信无关；select 可配 default 做非阻塞操作。

关键点：

  • select 只用于 channel 的发送/接收分支。
  • switch 是普通控制流，与并发通信无关。
  • select 可配 default 做非阻塞操作。

是否可以读写已关闭的 Channel

可直接说： 关闭后的 channel 可以读、不能写；向已关闭 channel 写会 panic。面试里可按三点展开：读关闭 channel：先读完缓冲，之后返回零值且 ok=false；写关闭 channel：立即 panic；重复关闭同一个 channel 也会 panic。

关键点：

  • 读关闭 channel：先读完缓冲，之后返回零值且 ok=false。
  • 写关闭 channel：立即 panic。
  • 重复关闭同一个 channel 也会 panic。

goroutine泄露的常见原因

  • 从 channel 里读，但是同时没有写入操作。但要注意：读已关闭 channel 不会阻塞；读 nil channel (未初始化的 channel)会永久阻塞。
  • 向 无缓冲 channel 里写，但是同时没有读操作
  • 向已满的 有缓冲 channel 里写，但是同时没有读操作
  • select操作在所有case上都阻塞()
  • goroutine进入死循环，一直结束不了

可见，很多都是因为channel使用不当造成阻塞，从而导致goroutine也一直阻塞无法退出导致的。

原子操作和锁的区别

可直接说： 原子操作适合简单共享变量读改写，锁适合保护一段临界区和复合状态。面试里可按三点展开：原子操作开销小，但表达能力有限；锁可保护多变量一致性与复杂业务逻辑；选型看“临界区复杂度”，不是只看性能。

关键点：

  • 原子操作开销小，但表达能力有限。
  • 锁可保护多变量一致性与复杂业务逻辑。
  • 选型看“临界区复杂度”，不是只看性能。

什么是自旋锁与互斥锁

可直接说： 自旋锁拿不到锁时忙等，互斥锁拿不到锁时挂起等待。面试里可按三点展开：自旋适合临界区非常短、竞争轻的场景；互斥锁适合临界区较长，避免 CPU 空转；Go 用户层主要直接用 sync.Mutex，底层会结合自旋策略优化。

关键点：

  • 自旋适合临界区非常短、竞争轻的场景。
  • 互斥锁适合临界区较长，避免 CPU 空转。
  • Go 用户层主要直接用 sync.Mutex，底层会结合自旋策略优化。

讲一下GMP模型，各个字母分别代表什么

GMP 各字母含义

  • G = goroutine，也就是 go 语言的协程执行单元。
    它保存函数栈、程序计数器、状态（_Grunnable/_Grunning/_Gwaiting/...）、defer/panic 等上下文。
  • M = machine，底层是操作系统线程（pthread）。
    真正执行代码的是 M。
  • P = processor，能够调度上下文/令牌。
    P 持有本地可运行队列、缓存（如 mcache）等资源。M 必须绑定 P 才能运行 Go 代码。

调度机制

首先，每一个P会维护一个队列，长度是256，当产生一个G时，会优先存放在自己P的本地队列里。

如果本地队列已经满了，就会将前一半的G取出，存放在全局队列，这样新产生G时，就又会存放在自己的本地队列，这样也符合局部性的原则。

如果P里面没有G，它会先去全局队列里拿（这个操作需要加锁）。如果全局队列也没有，它会从IO就绪队列里拿。如果还没有它又会从其他P队列窃取，这也就是work-stealing机制，根据源码，最多进行约 4 轮窃取尝试。

还有一个比较著名的机制Hand Off机制，如果P中的某个G陷入内核的阻塞态，那么这个P会和M做一个Hand Off，这样就不耽误这个P上的其他G运行。

P主要有2个作用，第一个是它把M和本地队列做一个解耦合，如果没有P这一层，那么如果本地队列的某一个G陷入内核阻塞态，那其它G都要跟着一起阻塞，没有办法做到真正的并行。还有一个是它有自己的本地队列，因此不需要一直加全局锁，从而减少锁的力度。它其实实现了各个P队列之间的负载均衡，充分利用了CPU的资源。

Golang GC 三色标记法

初始时对象逻辑上分为白/灰/黑三类：

  • 白色：尚未发现，理论上可回收候选。
  • 灰色：已发现但其引用还没扫描完。
  • 黑色：对象本身及其引用都扫描完成。

在并发三色标记（以 Go 为例）里，通常会看到 2 次 STW，场景分别是：

  1. 初始标记（Initial Mark）STW

    发生时机：GC 刚开始。

    做的事：

    • 暂停所有用户 goroutine（mutator）。
    • 扫描根对象（全局变量、栈、寄存器 可达对象）。
    • 把根可达对象放入标记流程（灰/黑队列）。
    • 打开写屏障，切到并发标记阶段。
      目的：拿到一个一致的“起始快照”，让后续并发标记可安全进行。

  2. 重新标记 / 标记终止（Remark / Mark Termination）STW

    发生时机：并发标记快结束时。

    做的事：

    • 再次短暂停顿，处理并发期间遗留的标记工作（灰对象队列、写屏障缓冲）。
    • 保证不存在“黑对象直接指向白对象”的漏标风险。
    • 完成标记闭环，准备进入清扫阶段。
      目的：在回收前做最终一致性收敛，避免误回收活对象。

混合写屏障规则

三色标记在并发下的核心风险是“漏标”

  1. 用户把一个白对象挂到黑对象下面（黑 -> 白），
  2. 同时又把原来能到达这个白对象的那条引用删掉，导致它不再能从灰对象链路被扫描到。
  • 这样白对象其实还活着，但 GC 看不到它，最后会被误回收（错误回收 / 悬挂指针风险）。

因此需要混合写屏障

具体操作:

1、GC开始将栈上的对象全部扫描并标记为黑色(之后不再进行第二次重复扫描，无需STW)，

2、GC期间，任何在栈上创建的新对象，均为黑色。

3、被删除的对象标记为灰色。

4、被添加的对象标记为灰色。

  • Go 1.8+ 的混合写屏障把“栈重扫”这件事基本消掉了，所以第二次 STW 的主要成本显著降低。
  • 但从 GC 周期视角看，仍有“开始”和“结束”这两段短 STW（只是更短了）。
